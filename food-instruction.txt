NutritionProfile je 1:1 na tracker. To je správně, jeden tracker reprezentuje jednu „nutriční identitu“.

GoalPlan je časová vrstva. Tohle potřebuješ, jinak každá změna cíle zničí historii.

NutritionTarget je verzovaný snapshot doporučení. Bez něj nebudeš umět auditovat změny targetu.

DailyNutritionLog a DailyBodyMeasurementLog jsou oddělené schválně. Nesnaž se to slít do jedné tabulky jen proto, že oboje má datum. To je líný model, ne chytrý model.

WeeklyCheckin je materializovaný agregát. Ano, šlo by to počítat on-demand. Ne, není to vždy chytré. Jakmile budeš chtít historičnost algoritmu a stabilní snapshoty, budeš rád, že to máš uložené.

TdeeAdjustmentEvent je audit algoritmu. To není kosmetika, to je jádro té aplikace.

Tady je kritická poznámka: NutritionTargetService nemá samo počítat TDEE ani makra. Má orchestrace:

natáhne profil, plán, logy
zavolá calculation services
uloží snapshot
vrátí DTO

To je správně. Jakmile začneš cpát vzorce do service impl, za měsíc nepoznáš, co je use-case a co je matematika.

Co snapshotovat a co dopočítávat

Snapshotovat:

NutritionTarget
WeeklyCheckin
TdeeAdjustmentEvent

Dopočítávat on-demand:

acceptable macro ranges
warningy/reminders
chart projections
aktuální dashboard summary
BF, pokud chceš jen „preview“ bez uložení

Přepočítat a uložit:

BF při uložení DailyBodyMeasurementLog, protože to je auditovatelný derived fact
weekly snapshot při uzavření týdne nebo po změně záznamů v týdnu
nový nutrition target při adaptivní korekci

Neukládat:

stringy typu "150 - 190"
texty typu "Reminder: your calories..."
názvy dnů
měsíční summary řádky

Doporučený orchestration flow

Flow pro první setup:

upsert NutritionProfile
create GoalPlan
generate initial NutritionTarget

Flow pro denní log:

upsert DailyNutritionLog
pokud jde o konec týdne nebo explicitní recalc, vygenerovat WeeklyCheckin
pokud existuje dost týdnů, spustit AdaptiveNutritionAdjustmentService
uložit TdeeAdjustmentEvent
uložit nový NutritionTarget

Flow pro obvody:

upsert DailyBodyMeasurementLog
spočítat BF
uložit BF
pokud je relevantní týdenní checkin, přepočítat ho

Dobře. Teď konečně to podstatné, ne další katalog tříd. Níže máš implementace výpočetních služeb v pořadí, které dává smysl. Držím je co nejblíž tomu Google Sheetu, ne nějakému „lepšími lidmi vymyšlenému“ fitness modelu.

Zásady, které držím stejně jako sheet:

interně všechno v kg a cm
BF z obvodů podle US Navy rovnic, přesně podle toho, co máš v sheetu
baseline TDEE z Katch-McArdle
targetCalories = baselineTdee + dailyAdjustment
denní adjustment z týdenní změny váhy přes 7700 kcal / kg / 7
protein heuristika podle BF %
fat heuristika podle BF %
carbs jako zbytek
adaptive adjustment podle reálné změny vs očekávané změny
až od 4. checkpointu, stejně jako sheet naznačuje

Níže používám pár rozumných konstant. Když budeš chtít absolutní 1:1 shodu se sheetem i v detailech zaokrouhlování a smoothingu, doladíme to na základě těch schovaných sloupců typu AX, AY, AW. Teď ti dávám robustní jádro, které už je velmi blízko.

Praktické user flow od nuly

Teď úplně jednoduše, jak s tím user pracuje krok po kroku.

Scénář A: první setup

PUT /api/nutrition-profile/{trackerId}
User nastaví profil.
POST /api/goal-plan/{trackerId}
User založí plán.
POST /api/nutrition-target/{trackerId}/{goalPlanId}/initial
Systém spočítá první target.
GET /api/nutrition-target/{trackerId}/current
Frontend zobrazí aktuální doporučení.

To je základní onboarding.

Scénář B: běžné denní používání

GET /api/nutrition-target/{trackerId}/current
User vidí, kolik má jíst.
PUT /api/daily-nutrition-log/{trackerId}
User zapíše dnešní váhu a kalorie.
volitelně PUT /api/daily-body-measurement-log/{trackerId}
User zapíše obvody.
GET /api/nutrition-dashboard/{trackerId}
Frontend ukáže aktualizovaný graf a stav.

To je denní rutina.

Scénář C: konec týdne

user má za týden nalogovaná data
frontend zavolá POST /api/weekly-checkin/{trackerId}/{goalPlanId}/{weekIndex}
backend vytvoří týdenní snapshot
pokud už je dost týdnů, může vzniknout nový target
frontend zavolá GET /api/nutrition-target/{trackerId}/current
user vidí nový target

To je weekly maintenance flow.

Scénář D: změna režimu

user dokončí cut
založí nový plán přes POST /api/goal-plan/{trackerId}
aktivuje ho nebo je rovnou active
zavolá se POST /api/nutrition-target/{trackerId}/{goalPlanId}/initial
jede dál pod novým plánem

To je lifecycle mezi plány.

Scénář E: user chce ručně přepsat target

POST /api/nutrition-target/{trackerId}/{goalPlanId}/manual-override
backend založí nový target snapshot
GET /api/nutrition-target/{trackerId}/current
frontend ukáže override target

To je advanced flow.

Které endpointy jsou pro usera „core“
